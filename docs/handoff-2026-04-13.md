## What Was Built This Session (2026-04-13)

Feature: **Agent self-provisioning**. When the user says *"spin up a finance agent"*, the current agent now performs the full provisioning flow end-to-end.

### Architecture — four isolated units

#### 1. `SecretStore` (`app/secret_store.py`)
Thin `keyring` wrapper. Stores tokens in OS Keychain (macOS) or Credential Manager (Windows) keyed by `{agent}:{channel}`. `pyproject.toml` now depends on `keyring>=24`.

#### 2. `AccountConfig.token_ref` (`app/config.py`)
New optional `token_ref` and `agent` fields on `AccountConfig`. When an account has `token_ref: "finance:telegram"`, the loader resolves it via `SecretStore` at startup. Plaintext `token` still works for backward compat. iMessage accounts skip the token entirely and key off `chat_identifier`.

#### 3. Router hot reload (`app/router.py::add_account`)
New `add_account(account_id)` method:
- Reloads `config.json`
- Builds a new `AccountRuntime` for the target account
- Starts the channel (event-driven or polling)
- Registers a scheduler sender
- Spawns a worker thread, registered in `self._workers[account_id]`
- Thread-safe via `self._accounts_lock`

The bind_channel tool calls this so new bots come alive without a process restart.

#### 4. Provisioning tools (`app/agent_provisioning.py` + `app/tools.py`)

Three new tools registered when `build_default_registry` receives `agents_dir + config_path + router`:

- **`create_agent(name, persona, display_name?, description?)`** — copies `agents/_template/` to `agents/<name>/`, renders `{{display_name}}`/`{{persona}}`/`{{description}}` placeholders. Validates name (lowercase alnum+dash, 1-32 chars, not reserved).
- **`bind_channel(agent, channel, token?, app_token?, chat_identifier?)`** — validates credentials against the platform API (Telegram `getMe`, Discord `/users/@me`, Slack `auth.test`), stores tokens in `SecretStore`, appends account to `config.json`, calls `router.add_account()`. Atomic — any failure rolls back.
- **`list_imessage_chats(limit?)`** — reads `~/Library/Messages/chat.db` read-only, returns recent chats with `chat_identifier`. macOS only; errors cleanly on Windows.

#### 5. Universal agent knowledge (`app/context_builder.py`)
Added a **Spawning Sub-Agents** section to the behavioral directives injected into every agent's prompt. Every agent — default, spawned, or custom — knows about the provisioning tools from the first message, no per-agent AGENT.md edits needed.

#### 6. Template (`agents/_template/`)
New template folder (un-gitignored via `!agents/_template/`). Contains `agent.json`, `AGENT.md`, `TOOLS.md` with placeholders. Source for `scaffold_agent`.

## Channel provisioning flow (Telegram example)

```
User:    "spin up a finance agent with telegram"
Agent:   [create_agent(name="finance", persona="Tracks spending, investments, net worth.")]
         → "Agent 'finance' scaffolded at .../agents/finance/.
            Next step: bind_channel(agent='finance', channel='telegram', token='...')"
Agent:   "To bind Telegram: open @BotFather, /newbot, pick name + username, paste the token back."
User:    "Here: 1234:ABCabc..."
Agent:   [bind_channel(agent="finance", channel="telegram", token="1234:ABCabc...")]
         → validates token with Telegram getMe (gets @finance_bot)
         → SecretStore.set("finance", "telegram", token)
         → config.json: accounts[finance-telegram] = {platform: telegram, agent: finance, token_ref: "finance:telegram", ...}
         → router.add_account("finance-telegram") → new polling thread live
         → "Bound telegram channel to agent 'finance' (@finance_bot). Router hot-reloaded — new bot is live."
```

Discord/Slack/iMessage follow the same shape with their own manual setup steps.

## Non-goals (explicit)

- **WhatsApp**: skipped. Meta Business API too heavy; unofficial libs violate ToS and can ban your number.
- **Per-agent Apple IDs**: not possible. iMessage uses shared Apple ID, routes per-agent by `chat_identifier` (group-chat-per-agent pattern).
- **Automated bot account creation**: impossible. Telegram, Discord, Slack all require an interactive human step at their respective UIs. The agent walks the user through copy-paste instructions.
- **Agent deletion**: `remove_account` stub reserved but not implemented in v1.

## Security Notes

- Tokens are written to OS keyring **only** — plaintext `config.json` paths still work for backward compat but `bind_channel` always uses keyring
- `config.json` stores `token_ref: "agent:channel"` instead of the token itself, so accidental commits/backups/screen-shares don't leak credentials
- `list_imessage_chats` opens `chat.db` read-only via `file:...?mode=ro` SQLite URI; grants Full Disk Access are required to read it
- `create_agent` validates name against a strict regex, rejects `_template` reservation, refuses duplicates — no path traversal risk

## Known Follow-ups (parked)

1. **Agent deletion / `router.remove_account`** — interface reserved, not implemented. Add when the user needs to retire an agent.
2. **`list_imessage_chats` on Windows** — returns a clean error today; if multi-device iMessage provisioning ever matters, the macOS side can export a chat list JSON that the PC reads.
3. **Rollback on `router.add_account` failure** — currently the config write is persisted even if the hot reload fails; user gets a warning and the next runtime restart will pick it up. Could add full rollback if flakiness becomes a problem.
4. **Credential rotation** — no `rotate_channel_token` tool yet. Manual process today: call `bind_channel` again with new token, it overwrites the keyring entry.

## Bottom Line

Agents can now create agents. Each new agent gets its own isolated bot identity on every supported channel (Telegram, Discord, Slack, iMessage), tokens go to the OS keyring on day one, router hot-reloads without a restart, and the knowledge of how to do all this is injected into every agent's prompt so they know from the first message. 301 tests passing, no regressions. WhatsApp is excluded by design.
