## Feature 1: Message Forwarding

### Problem

Users cannot send a message from one surface (e.g., Telegram) to another surface (e.g., Discord or a different Telegram chat). The channel adapters exist and the scheduler already has registered send callbacks per surface, but there's no user-facing command.

### Goal

`/forward <target> <message>` sends a message to another registered surface/chat. Uses the scheduler's existing send callback infrastructure.

### Design

#### Command: `/forward <target> <message>`

Target format: `<surface>:<account_id>:<chat_id>` or shorthand `<chat_id>` (uses same surface/account).

Examples:
- `/forward telegram:primary:99999 Hey, check this out`
- `/forward 99999 Quick note to another chat`

#### Implementation

In `app/commands.py`, add `/forward` command. The CommandHandler needs access to the scheduler's `_send_callbacks` dict to dispatch messages. The scheduler already has a public `_send_callbacks` — expose a `send_message(surface, chat_id, text)` method on Scheduler instead of accessing internals.

**Scheduler addition** (`app/scheduler.py`):
```python
def send_to(self, surface: str, chat_id: str, text: str) -> None:
    """Send a message to a specific surface/chat via registered callback."""
    callback = self._send_callbacks.get(surface)
    if callback is None:
        raise SchedulerError(f"No sender registered for surface: {surface!r}")
    callback(surface, chat_id, text)
```

**Command handler** (`app/commands.py`):
- Parse target and message from `/forward` input
- Full format: `surface:account_id:chat_id` → use that surface key
- Short format: just `chat_id` → use current `surface` (from handle() args)
- Call `self._scheduler.send_to(surface_key, chat_id, message)`
- Return confirmation or error

#### Error Handling

- Missing scheduler → "Scheduler not available."
- Invalid target format → usage help
- No registered sender for target surface → "No sender registered for surface: X"
- Send failure → "Failed to forward: {error}"

### What Is Not Changing

- Scheduler internals — just adding a public method
- Channel adapters — unchanged
- Router — unchanged (CommandHandler already receives scheduler)

## Feature 2: Conversation Search/Export

### Problem

Users can view recent transcript entries with `/transcript [n]` but cannot search conversation history or export it. No full-text search across transcripts exists.

### Goal

1. `/search-chat <query>` — search current chat's transcript for keyword matches
2. `/export` — export current chat's full transcript as formatted text

### Design

#### `/search-chat <query>`

Search the current chat's transcript JSONL file for entries matching the query.

**MemoryStore addition** (`app/memory.py`):
```python
def search_transcript(
    self, surface: str, chat_id: str, query: str,
    *, account_id: str = "primary", agent_name: str = "main", limit: int = 10,
) -> list[TranscriptEntry]:
    """Search transcript entries matching query (case-insensitive substring)."""
```

- Read all lines from the transcript JSONL file
- Filter entries where `query.lower()` appears in `message_text.lower()`
- Return last `limit` matching entries (most recent first)

**Command** (`app/commands.py`):
- Parse query from `/search-chat <query>`
- Call `memory_store.search_transcript(...)`
- Format results like `/transcript` does

#### `/export`

Export the current chat's transcript as formatted text.

**Command** (`app/commands.py`):
- Read all transcript entries for current surface/chat/agent
- Format as readable text: `[timestamp] You/Assistant: message`
- Return the text (Telegram will handle splitting long messages)
- Cap at last 200 entries to avoid huge payloads

### What Is Not Changing

- Dashboard API — unchanged (already has its own endpoint)
- Embeddings — not used for transcript search (simple keyword is fine for now)
- Transcript storage format — unchanged
