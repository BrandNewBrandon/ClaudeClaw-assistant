## Key Architecture Notes

### Message flow
1. `AssistantRouter.run()` starts one polling thread per account
2. `channel.get_updates()` → `ChannelMessage` objects
3. `_handle_message_locked()` — per-chat lock:
   - Auth check (or DM pairing) → slash command check → cache/cooldown → compaction → context load → `_generate_reply_with_tools()`

### Two execution paths in `_generate_reply_with_tools`
- **Streaming** (Telegram): sends `▌` placeholder, edits in place as tokens arrive, throttled 300ms, tool calls show status lines
- **Blocking** (Discord, Slack, fallback): `subprocess.run()`, full response, up to 3 tool iterations

### Claude CLI invocation
- Prompt passed via stdin (`-p -`) to avoid Windows command-line length limits
- Both `subprocess.run()` (blocking) and `subprocess.Popen` (streaming) use this pattern

### Response cache
- Key: `(chat_id, agent_name, normalized_message)`
- TTL: configurable (default 300s), max 500 entries with oldest-first eviction

## Privacy / Safety Note

Everything runs locally. Conversation history, memory, config, and tool execution never leave the machine. The only external data flow is the message → Anthropic API call. No ClaudeClaw server, no third-party backend, no sync service.
