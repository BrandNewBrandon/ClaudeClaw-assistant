## Key Architecture Notes

### Message flow
1. `AssistantRouter.run()` starts one polling thread per account
2. `channel.get_updates()` ‚Üí `ChannelMessage` objects
3. `_handle_message_locked()` ‚Äî per-chat lock:
   - Auth check (or DM pairing) ‚Üí slash command check ‚Üí cache/cooldown ‚Üí compaction ‚Üí context load ‚Üí `_generate_reply_with_tools()`

### Two execution paths in `_generate_reply_with_tools`
- **Streaming** (Telegram): sends `‚ñå` placeholder, edits in place as tokens arrive, throttled 300ms, tool calls show status lines
- **Blocking** (Discord, Slack, fallback): `subprocess.run()`, full response, up to 3 tool iterations

### Claude CLI invocation
- Prompt passed via stdin (`-p -`) to avoid Windows command-line length limits
- Both `subprocess.run()` (blocking) and `subprocess.Popen` (streaming) use this pattern

### Tool registry
- `build_default_registry()` in `app/tools.py` registers 8 tools: `web_search`, `web_fetch`, `read_file`, `write_file`, `list_dir`, `disk_usage`, `list_processes`, `run_command`
- `run_command` is gated behind YES/NO approval; all others run freely
- `safe_commands` in `agent.json` bypasses the approval gate for matching command prefixes

### Response cache
- Key: `(chat_id, agent_name, normalized_message)`
- TTL: configurable (default 300s), max 500 entries with oldest-first eviction
