## What Was Completed This Session (2026-04-12)

### 1. `run_command` hardening ŌĆö failed native builds no longer freeze the assistant

**Problem:** On Windows, an assistant user asked the bot to `pip install py-rust-stemmers`. The package needs Visual Studio Build Tools to compile Rust code; the machine had no MSVC linker. pip ran Rust compilation for ~30s, dumped 58 lines of compiler output (with garbled double-decoded emoji bytes like `├░┼Ė"┬”`), then failed. The assistant stopped responding.

**Root cause:** Three compounding issues in `app/tools.py::_run_command`:
- 60s timeout was enough to *start* but too short for typical native-extension installs
- Huge compiler stderr got passed back to Claude as tool result, bloating the prompt
- Non-ASCII garbled bytes confused the model into producing an empty response, hitting the "Claude returned no text" path

**Fix (`app/tools.py`):**
- `pip install` / `npm install` / `cargo install` / `apt|brew|choco install` ŌåÆ timeout raised to 300s
- On failure (`returncode != 0`) with stderr > 30 lines ŌåÆ keep only the last 30 lines, prefix with `...[earlier output truncated]` (real error survives, noise is dropped)
- Non-ASCII characters stripped from tool output with `re.sub(r"[^\x20-\x7E\n\r\t]", "", output)` to prevent model confusion

**Fix (`app/assistant_cli.py::_cmd_update`):**
- Before the reinstall, run `pip cache purge` (30s timeout) so `assistant update` always starts with a clean build cache. This means any half-compiled Rust/C++ artifact from a prior failed install gets cleared automatically next update.

**Tests (`tests/test_tools.py`):**
- `test_run_command_truncates_stderr_on_failure` ŌĆö 50 lines of noise + 1 real error ŌåÆ only real error survives
- `test_run_command_strips_non_ascii` ŌĆö garbled bytes removed, ASCII preserved
- `test_run_command_uses_longer_timeout_for_pip` ŌĆö 300s for pip, 60s for echo

### 2. Silent-reply placeholder cleanup

**Problem:** When an agent's final reply is the `__SILENT__` sentinel (suppress the response), the streaming path just skipped sending. But if an earlier tool-status bubble was already shown (e.g. `ŌÜÖ’ĖÅ Running commandŌĆ”`), that message was left dangling in the chat as a ghost.

**Fix (`app/telegram_client.py`):**
- New `delete_message(chat_id, message_id)` ŌĆö calls Telegram's `deleteMessage` API, silently swallows errors (already-deleted is harmless)

**Fix (`app/channels/base.py`):**
- `BaseChannel.delete_message()` ŌĆö default no-op so other channels don't need to implement it

**Fix (`app/channels/telegram.py`):**
- Wires the channel adapter through to `TelegramClient.delete_message`

**Fix (`app/router.py`):**
- In `_run_streaming_tool_loop`, when `last_output.strip() == "__SILENT__"` and `message_id_box[0] is not None`, call `channel.delete_message()` so the leftover placeholder disappears instead of hanging in the chat

### 3. Python 3.14 guard ŌĆö `assistant update` no longer silently explodes on unsupported interpreters

**Problem:** The Windows user's `.venv` was built on Python 3.14. `py-rust-stemmers` (a transitive dep of fastembed/BM25 search) has no prebuilt wheels for 3.14, so pip tried to compile from Rust source ŌåÆ needed MSVC ŌåÆ failed. The first `run_command` fix didn't help because this failure happened **inside `assistant update` itself**, not during a tool call. Pulling the fix just ran into the same wall again.

**Fix (`pyproject.toml`):**
- `requires-python = ">=3.11,<3.14"` ŌĆö pip refuses install on 3.14+ with a clean error message, instead of attempting a Rust compile that will fail

**Fix (`app/assistant_cli.py::_cmd_update`):**
- Before running `pip install`, probe the venv's Python version by executing `python -c "import sys; ..."` inside `.venv/Scripts/python.exe` (Windows) or `.venv/bin/python` (Unix)
- If the venv Python is outside `[3.11, 3.14)`, print a clear error with the exact rebuild commands for the platform and abort with exit code 1
- **Does not auto-delete the venv** ŌĆö that's destructive and the user should confirm. The error message includes copy-paste-ready commands.

Error output looks like:
```
Error: your .venv uses Python 3.14, which is not supported.
  ClaudeClaw requires Python 3.11, 3.12, or 3.13.
  Python 3.14 is too new ŌĆö many native-extension wheels are not yet published for it.

To fix, rebuild the .venv on a supported Python:
  cd C:\Users\ace09\ClaudeClaw\assistant
  rmdir /s /q .venv
  py -3.12 -m venv .venv
  assistant update
```

## Key Architecture Notes (unchanged from prior handoff, recap)

### Message flow
1. `AssistantRouter.run()` starts one polling thread per account
2. `channel.get_updates()` ŌåÆ `ChannelMessage` / `ChannelCallback` objects
3. `_handle_message_locked()` per-chat lock ŌåÆ auth / pairing ŌåÆ slash-command ŌåÆ cache/cooldown ŌåÆ compaction ŌåÆ context load ŌåÆ `_generate_reply_with_tools()`

### Two execution paths in `_generate_reply_with_tools`
- **Streaming** (Telegram): `Ō¢ī` placeholder lazily sent on first real chunk, edited in place, 300ms throttle. Tool calls replace the bubble with a friendly status line. **Silent replies now delete the bubble if one exists.**
- **Blocking** (Discord, Slack, fallback): `subprocess.run()`, full response, up to 3 tool iterations (10 with computer use)

### Tool registry
- `build_default_registry()` in `app/tools.py` registers 11 tools: `web_search`, `web_fetch`, `read_file`, `write_file`, `list_dir`, `disk_usage`, `list_processes`, `run_command`, plus (optionally) computer-use trio
- `run_command` is gated behind YES/NO approval (or inline Approve/Deny buttons on Telegram); `safe_commands` in `agent.json` bypasses
- **New as of today:** `run_command` has package-install-aware timeout, stderr tail truncation on failure, non-ASCII stripping
